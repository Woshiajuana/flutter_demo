- const 定义的变量是编译时常量
- final 定义的变量是运行时常量

## 三棵树渲染机制
- Widget：对视图的结构化描述，存储视图渲染相关的配置信息：布局、渲染属性、事件响应等信息。
- Element：Widget的实例，是渲染树的节点，负责将Widget渲染到屏幕上。
- RenderObject：负责实际的渲染工作，将Widget的配置信息转换为屏幕上的像素。

## 渲染过程
1. 通过 Widget 树生成对应的 Element 树；
2. 创建响应的 RenderObject 并关联到 Element.renderObject 属性上；
3. 构建 RenderObject 树，深度优先遍历，确定树中各对象的位置和尺寸(布局)，把他们绘制到不同图层上；
4. Skia 在 Vsync 信号同步时直接从渲染树合成 Bitmap，最后交给GPU渲染；

## Element 树的优点
- 高效渲染：将 Widget 树的变化(diff)，只将真正需要修改的部分同步到 RenderObject 树
- Element 是可复用的

## Widget 的四个直接子类
- StatelessWidget：无状态的 Widget，其状态是不可变的。
- StatefulWidget：有状态的 Widget，其状态是可变的。
- ProxyWidget：代理 Widget，用于在 Widget 树中添加额外的功能或属性。
  - 例如：InheritedWidget、ParentDataWidget 等。
- RenderObjectWidget：用于构建复杂或底层渲染相关的自定义 Widget

## 分层架构图
- Embedder（嵌入/平台层）：负责与底层操作系统进行交互，提供渲染、输入、辅助功能等服务，并且管理事件循环队列。
- Engine（引擎层）：提供Flutter核心API的底层实现，包括图形绘制、文本布局、文件和网络IO、插件架构和Dart运行时和编译环境。
- Framework（框架层）：负责提供上层API的封装，用于构建应用。

## Flutter 线程模型
- Platform Task Runner：主Task Runner
- UI Task Runner：负责执行Dart代码
- Raster Task Runner：将UI Task Runner 生成的 layer tree 光栅化 为 bitmap，然后显示在屏幕上。它需要访问设备的 rasterizer (通常由GPU提供)，所以它也被称为 GPU Task Runner。
- IO Task Runner：负责执行IO相关的任务，如文件读写、网络请求、图片解码等，它可以创建和管理其它Dart isolate，以便在后台执行一些复杂的计算。
